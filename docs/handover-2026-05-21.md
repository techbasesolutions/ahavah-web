## 6. Challenges, failures, patterns, lessons (READ THIS — it will save you cycles)

### 6.1 The email took FOUR iterations before it was accepted. Don't repeat this.
- **What went wrong:** I kept *rebuilding the email from scratch* (web-safe fonts, an invented dark "remnant" hero, a dark logo band) instead of *porting the canonical template's actual markup + exact styles*. The user said, in order: "fonts didnt carry over" → "doesnt look like the source of truth" → "Again this looks nothing like any of the email templates."
- **The lesson:** When the user hands you a Claude Design bundle, the deliverable is **pixel-faithful recreation of the canonical**, not your interpretation. Read the canonical's exact CSS values (`emails/canonical/Ahavah-Email-Templates.html`) and reproduce them. For email specifically: **inline-style every element** (Gmail strips `<style>`) AND include a `<style>` block + Google Fonts `<link>` + a `prefers-color-scheme` dark block (for Apple Mail/iOS). The final `base.py` does exactly this.
- **Gmail limits you can't fix:** Gmail (web + Android) **strips web fonts** → Ultra falls back to Georgia there; and Gmail **force-inverts** in dark mode → a dark-ink logo on white vanishes. Mitigations: `color-scheme`/`supported-color-schemes` meta (reduces Gmail's aggressive inversion) + a light/dark **logo swap** (`<img class="e-logo-light">` + `<img class="e-logo-dark">` toggled by the dark media query). Apple Mail/iOS render the full canonical (fonts + light design); set expectations accordingly.

### 6.2 The feature slider took TWO misses (images, phone-frame) before "editorial type slides".
- Showing raster *pictures of the design* (the marketing composites) and then wrapping them in a phone frame both got rejected ("that won't do at all"). The accepted answer: **rebuild the slides as live kit components**. The canonical "slides" were always live DOM (the PNGs were just renders).
- **Lesson (now a standing memory rule `[[feedback_new_surfaces_frontend_design]]`):** EVERY new surface/section/element — *including emails* — must be designed via `/frontend-design` (+ `/ui-implementer`, `/mobile-responsive`) using **kit primitives only**, then deployed. No raster stand-ins, no hand-rolled atoms.

### 6.3 "Dating" is the wrong frame. The Torah-observant frame is **courtship and marriage / matchmaking**.
- The user first supplied taglines using "dating" verbatim, then reversed: *"dating is not a thing in the torah observant world its courtship and marriage."* Use **courtship, marriage, matchmaking, betrothal** — never frame Ahavah itself as "dating" (you may contrast against secular/Christian *dating*).

### 6.4 Fabricated numbers were live in the codebase. Remove them on sight.
- The hero had a hardcoded **"12,400+"** waitlist stat and a fabricated **"63 countries"**; the CTA band + sticky CTA also had "12,400+". All removed (replaced by the live count with floor, "Worldwide", "Founding members welcome"). The real count is **2**. Never display invented metrics (`[[feedback_no_fabrication]]`).

### 6.5 Strict ESLint will block your commits (lint-staged pre-commit hook).
- `no-restricted-syntax` bans **arbitrary Tailwind values** (`leading-[1.55]`, `text-[13px]`) and **inline sizing/spacing styles** (`style={{ fontSize }}`). Pages doing brand display typography (`page.tsx`, `privacy`, etc.) carry a file-level `/* eslint-disable no-restricted-syntax */` with a justification comment — match that pattern for new files that need it. `background:` inline is allowed; `fontSize`/`width` inline is not.
- `react-hooks/set-state-in-effect`: setting state synchronously in `useEffect` is flagged. Either disable-next-line (matching `waitlist/page.tsx`) or collapse to a single `setState`. An **unused** `eslint-disable` directive is itself an error under `--max-warnings=0`.
- Run `npx eslint <files>` before committing to avoid a failed hook. The hook stashes + restores on failure.

### 6.6 Em dashes are banned in user-facing strings.
- `[[feedback_no_em_dashes]]`. Use colons / periods / "and". `&middot;` (·) is fine; `&mdash;` (—) is not. Re-scan generated copy before shipping.

### 6.7 Deploy/verify mechanics that bit me
- **Bash cwd resets** to `d:\Antigravity` between calls → `cd` explicitly.
- **API branch is `ahavah/main`**, not `main`; `git push origin main` silently no-ops ("Everything up-to-date").
- `docker compose ... exec` on the droplet fails (blank `${VAR}`s) → use `docker exec ahavah-api-api-1`.
- The `gh run watch "<id>" --exit-status` pattern is reliable for the droplet deploy; get the id via `gh run list --branch ahavah/main --workflow "Deploy ahavah/main → droplet" --limit 1 --json databaseId -q '.[0].databaseId'`.
- Vercel `ls` polling via `sed -n '7p'` is flaky (timing); just re-list rows `6,9p` to read the current state.
- `publish.yml` GHA on `ahavah/main` fails independently of the deploy (it's a separate CI; the deploy workflow "Deploy ahavah/main → droplet" is the one that matters).

### 6.8 Process that worked
- The superpowers loop (brainstorming → writing-plans → executing-plans) on the waitlist-growth feature produced a clean spec + plan + 8 committed tasks. Use `AskUserQuestion` for genuine forks (it saved a wrong build on the IG-template + count + scope decisions). Gate outward-facing/irreversible actions (email sends) behind explicit confirmation.
